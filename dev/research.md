## 2. 핵심 규칙

### 2.1 게임판 구성
- **17×10** 격자에 **1~9** 숫자가 적힌 사과 **170개** 배치
- 숫자 분포는 랜덤 — 단, **판 전체 사과의 합이 10의 배수**가 되도록 생성됨

> **구현 연결**
> - `COLS = 17`, `ROWS = 10`, `TOTAL_APPLES = 170` (board.h)
> - `board_init()`의 `do-while`에서 `sum % 10 != 0` 조건으로 구현

### 2.2 기본 조작
1. 화면을 **드래그**해서 박스(사각형 영역)를 그린다
2. 박스 안에 들어온 사과 숫자의 합이 **정확히 10**이 되면 박스가 **빨간색**으로 변함
3. 드래그를 **해제**하면 해당 사과들이 제거되고 **사과 1개당 1점** 획득
4. 합이 10이 아닌 상태로 해제하면 아무 일도 일어나지 않음

> **구현 연결**
> - 박스 그리기: `draw_drag_box()` (render.c) — Level 04
> - 빨간색 전환: `draw_drag_box`에 `sum == 10`이면 RED 색상 — Level 05
> - 드래그 해제 감지: `IsMouseButtonReleased` (input.c) — Level 06
> - 사과 제거: `board_remove_in_rect()` (board.c), 점수: `game->score += removed` — Level 06

### 2.3 제한 시간
- **2분 (120초)**
- 화면 우측 게이지로 잔여 시간 확인 가능
- 제한 시간 내 모든 사과 제거 시 **올 클리어** — 소요 시간이 기록됨

> **구현 연결**
> - `game->timeLeft = 120.0f` 로 초기화 (game.c)
> - `timer_update()`에서 `GetFrameTime()`을 이용해 감소 (timer.c) — Level 07
> - `draw_timer_bar()` 로 우측 게이지 표시 (render.c) — Level 07
> - `check_all_cleared()`로 올 클리어 판정, `clearTime = 120.0f - timeLeft` (timer.c) — Level 07

### 2.4 최대 점수
- 사과 170개 전부 제거하면 이론상 최대 **170점**

## 3. 합이 10이 되는 조합

### 2개 조합 (5가지)
```
1+9 / 2+8 / 3+7 / 4+6 / 5+5
```

### 3개 조합 (8가지)
```
1+1+8 / 1+2+7 / 1+3+6 / 1+4+5
2+2+6 / 2+3+5 / 2+4+4 / 3+3+4
```

### 4개 조합 (9가지)
```
1+1+1+7 / 1+1+2+6 / 1+1+3+5 / 1+1+4+4
1+2+2+5 / 1+2+3+4 / 1+3+3+3
2+2+2+4 / 2+2+3+3
```

### 5개 이상 조합
- 이론상 존재하나 실전에서는 거의 사용하지 않음

> **구현 연결**
> - 이 조합들은 별도로 코드에 하드코딩하지 않는다. `board_sum_in_rect()`가 범위 내 사과를 전부 더해서 합을 구하므로, 자연스럽게 모든 조합이 처리된다
> - 단, **드래그 박스는 사각형** 영역만 선택 가능. 불규칙한 모양(L자형 등)은 원작에서도 불가

## 4. 전략 및 난이도 분석

### 4.1 클리어 불가능 조건
- **9**는 **1**과만 짝이 될 수 있음
- `9의 개수 > 1의 개수` 인 판은 구조적으로 올 클리어 불가능
- 이런 경우 **리셋(새 판)** 하는 것이 유리

> **구현 연결**
> - `board_init()`의 do-while 조건: `count_value(values, 9) > count_value(values, 1)` 이면 재생성
> - 이 조건 덕분에 우리 구현에서는 이론적으로 올 클리어 불가능한 판이 생성되지 않는다

### 4.2 판 리셋 전략
- 높은 숫자(특히 9, 8)가 과도하게 많이 나온 판은 리셋 고려
- 낮은 숫자 비율이 높을수록 다양한 조합이 가능해 유리

### 4.3 점수 효율
- **큰 묶음보다 작은 묶음**이 점수에 유리하지 않음 — 개수 기반 1점/사과이므로 묶음 크기는 무관
- 박스 안의 빈 공간에 포함된 사과도 합산에 포함됨에 주의

> **구현 연결**
> - 점수는 제거된 사과 개수(`removed_count`)만큼 올라간다 (`game->score += removed`)
> - 드래그 박스 내 빈 공간의 사과도 `board_sum_in_rect`가 포함해서 계산하므로, 의도치 않은 사과까지 포함될 수 있다. 플레이어가 직접 박스 크기를 조절해야 한다

## 5. UI 및 특수 기능

### 5.1 색상 설정
- 하단 **「薄い色 (연한 색)」** 버튼으로 사과 색을 연하게 변경 가능
- 장시간 플레이 시 눈 피로도 감소 목적

> **구현 연결**
> - `game.thinColor` bool 필드 (game.h)
> - `draw_apple()`의 `thin_color` 파라미터 (render.c) — Level 03
> - T키 토글, 화면 버튼 — Level 09

### 5.2 배경음악(BGM)
- BGM이 시간 경과 지표 역할을 함
- **6번째 BGM**이 재생되면 약 **12초 남은 시점** — 마무리 신호로 활용 가능

> **구현 연결**
> - Raylib의 `Music` 타입으로 BGM 구현 (audio.c) — Level 10
> - 원작의 BGM 파일을 직접 쓸 수는 없으므로 무료 BGM으로 대체

### 5.3 개발자 콘솔 조작 (비공식)
- `F12` → 콘솔에서 `timeAll = <초>` 입력으로 제한 시간 변경 가능
- 기본값: `timeAll = 120`

> **개발 팁 연결**
> - 우리 구현에서도 비슷하게 테스트 가능: `game.c`의 `game_init`에서 `game->timeLeft = 5.0f;`로 임시 변경. Level 07 참고

## 8. 게임의 중독성 구조 분석

- **단순한 규칙** (합이 10 = 제거) + **무한 반복 가능한 구조**
- **실력 요소**: 빠른 계산력, 시야 확보, 효율적 박스 배치
- **운 요소**: 초기 판 숫자 배치 (클리어 가능 여부 자체가 운에 달림)
- 실력과 운이 균형 있게 작용해 반복 도전 욕구를 자극함

> **개발 관점 메모**
> - "운 요소"를 완전히 제거하기 위해 board_init에서 클리어 불가 판을 걸러낸다 (4.1절 조건)
> - "실력 요소"는 구현이 아니라 플레이어에게 달린 부분 — 우리가 만드는 것은 도구(게임 로직)이고, 실력은 플레이어가 발휘한다
